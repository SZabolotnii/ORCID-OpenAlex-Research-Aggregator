# План тестування ResearchIQ

Дата: 2026-04-24

## 1. Мета

Цей план описує, як перевіряти ResearchIQ після змін у:

- multi-tenant авторизації
- серверному збереженні даних
- AI/chat/report generation
- інтеграції `OpenRouter`
- фронтенд-рефакторингу
- deployment/update на VPS

Мета тестування:

- підтвердити, що критичні користувацькі сценарії працюють стабільно
- зловити регресії після рефакторингу або backend changes
- перевірити ізоляцію tenant-даних і рольову модель `viewer/admin`
- переконатися, що AI-функції працюють із правильним LLM provider

## 2. Обсяг тестування

У межах цього плану перевіряються:

- фронтенд SPA
- backend API
- tenant auth/session flow
- CRUD для faculty
- імпорт даних
- chat
- report generation
- smoke-перевірка production/VPS середовища

Поза межами цього плану:

- глибоке навантажувальне тестування
- security audit сторонніми інструментами
- повноцінний e2e automation suite
- performance profiling великих tenant-датасетів

## 3. Рівні перевірки

### 3.1. Статичні перевірки

Виконуються локально перед ручним тестуванням:

```bash
npm run lint
npm run typecheck
npm run build
cd server && npm run build
```

Очікування:

- команди завершуються без помилок
- frontend build проходить успішно
- backend build проходить успішно

### 3.2. Локальний функціональний smoke

Виконується проти локально запущеного API:

```bash
APP_BASE_URL=http://localhost:3001 \
SMOKE_TENANT_ID=default \
npm run smoke
```

Для private tenant:

```bash
APP_BASE_URL=http://localhost:3001 \
SMOKE_TENANT_ID=default \
SMOKE_PRIVATE_TENANT_ID=your-private-tenant \
SMOKE_VIEWER_PASSWORD=viewer-password \
SMOKE_ADMIN_PASSWORD=admin-password \
npm run smoke
```

### 3.3. Ручне функціональне тестування

Проводиться в браузері після локального smoke або після деплою.

### 3.4. Production smoke

Проводиться після оновлення на VPS.

Детальний operational checklist: [production-smoke-checklist.md](/Users/docua/Project/Research/ORCID-OpenAlex-Research-Aggregator/docs/production-smoke-checklist.md:1)

## 5. Передумови

Перед запуском тестів потрібно:

- мати запущений frontend і backend або deployed VPS
- мати хоча б один public tenant
- мати хоча б один private tenant
- мати `viewer` і `admin` пароль для private tenant
- мати валідний `server/.env`
- для перевірки `OpenRouter` мати `OPENROUTER_API_KEY`

## 6. Тестові набори

## 6.1. Health та конфігурація

### TC-01. Health endpoint

Кроки:

1. Відкрити `GET /api/health`

Очікування:

- статус `200`
- тіло містить `{ "status": "ok" }`

### TC-02. Tenant config

Кроки:

1. Викликати `GET /api/tenant/:id/config`

Очікування:

- повертається `id`
- повертається `name`
- коректно відображається `public`
- коректно відображається наявність admin password

## 6.2. Авторизація та ізоляція tenant-ів

### TC-03. Public tenant доступний без логіну

Кроки:

1. Відкрити public tenant
2. Перейти на dashboard

Очікування:

- дані завантажуються без password prompt

### TC-04. Private tenant блокує анонімний доступ

Кроки:

1. Відкрити private tenant без активної сесії

Очікування:

- показується password flow
- запити на protected data без токена отримують `401`

### TC-05. Viewer login

Кроки:

1. Увести viewer password
2. Завантажити dashboard і faculty list

Очікування:

- логін успішний
- дані читаються
- CRUD-контроли недоступні
- `/search` недоступний

### TC-06. Admin login

Кроки:

1. Увести admin password
2. Перейти до `faculty`, `search`, `settings`

Очікування:

- логін успішний
- CRUD доступний
- protected write endpoints працюють

### TC-07. Logout

Кроки:

1. Увійти як admin
2. Виконати logout

Очікування:

- auth token очищається
- для private tenant доступ знову обмежується
- для public tenant лишається read-only доступ

## 6.4. Batch import

## 7. Негативні сценарії

- неправильний viewer password
- неправильний admin password
- спроба запису без токена
- спроба viewer виконати admin action
- пошкоджений JSON/CSV/template file
- порожній tenant
- tenant із дуже малою кількістю даних
- недоступний OpenRouter або Gemini provider

Очікування:

- система показує контрольовану помилку
- UI не ламається
- сервер не віддає неавторизований доступ

## 8. Критерії проходження

Реліз можна вважати придатним до rollout, якщо:

- `npm run lint` проходить
- `npm run typecheck` проходить
- `npm run build` проходить
- `server/npm run build` проходить
- `npm run smoke` проходить для цільового середовища
- P0-сценарії проходять вручну
- chat і reports працюють із цільовим LLM provider

1. `npm run lint`

3. `npm run build`
4. `cd server && npm run build`
5. `npm run smoke`
6. вручну перевірити:
