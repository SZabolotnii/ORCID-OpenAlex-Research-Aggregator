# Аналіз поточного стану проєкту ResearchIQ

Дата: 2026-04-23

## Короткий висновок

Проєкт перебуває у стані сильного MVP / pre-production прототипу. Архітектура вже достатньо зріла: є окремий фронтенд і бекенд, multi-tenant модель, AI-агент на `pi-agent-core`, імпорт даних з ORCID/OpenAlex, генерація звітів та базова рольова модель. При цьому для production-рівня залишаються критичні прогалини в безпеці tenant-даних, довірі до клієнта, синхронізації CRUD-операцій і підтримуваності фронтенду.

Під час аналізу:

- фронтенд збирається успішно через `npm run build`
- бекенд збирається успішно через `cd server && npm run build`
- у фронтенді є попередження про великий bundle
- git-стан чистий, окрім некоміченого `AGENTS.md`

## Що вже зроблено добре

### 1. Архітектура бекенду

Бекенд виглядає найзрілішою частиною системи:

- окремий пакет у `server/`
- `strict` TypeScript у `server/tsconfig.json`
- логічне розділення на:
  - `index.ts` для API
  - `agent.ts` для конфігурації AI-агента
  - `tools.ts` для інструментів
  - `reportService.ts` для генерації звітів

Це хороша база для подальшого розвитку.

### 2. AI-шар

AI-частина спроєктована сильно:

- системний prompt формується з локального зрізу факультету
- є набір окремих інструментів для рейтингів, пошуку публікацій, порівняння кафедр, citation trends та OpenAlex lookup
- більшість питань може закриватися локальним контекстом без зайвих зовнішніх викликів

Для внутрішньої аналітичної системи це правильний напрямок.

### 3. Інтеграції даних

Сервіси ORCID/OpenAlex/merge винесені окремо:

- `services/orcidService.ts`
- `services/openAlexService.ts`
- `services/dataMergeService.ts`

Логіка дедуплікації публікацій виглядає адекватною для поточного масштабу.

### 4. Продуктова цінність

Навіть у теперішньому стані проєкт уже вирішує реальну задачу:

- імпорт faculty по ORCID
- enrichment метрик через OpenAlex
- dashboard
- AI-чат по локальних даних
- генерація звітів

Тобто це не "заготовка", а робочий продукт із реальною користю.

## Критичні проблеми

### 1. Сервер не захищає приватні tenant-дані належним чином

Це найважливіша проблема.

У `server/src/index.ts`:

- `GET /api/data/:tenantId` віддає дані без перевірки ролі
- `POST /api/data/:tenantId` дозволяє запис без справжньої авторизації
- backward-compatible `POST /api/data` теж записує без захисту

Наслідок:

- приватність tenant-ів зараз тримається переважно на логіці фронтенду
- будь-який клієнт, що знає endpoint, може напряму працювати з даними

Для multi-tenant production це блокер рівня `P0`.

### 2. Можна перезаписати admin password без підтвердження старого

У `POST /api/tenant/:id/set-password` перевірка `currentAdminPassword` відбувається лише якщо це поле передали. Якщо пароль уже існує, але поле не надіслати, зміна все одно проходить.

Наслідок:

- існуючий admin password tenant-а можна змінити без знання чинного пароля

Це теж `P0`.

### 3. CRUD фактично живе в localStorage, а не в серверному джерелі істини

У `App.tsx` після змін список `facultyList` зберігається в:

- `localStorage.setItem(\`faculty_data_${TENANT_ID}\`, ...)`

Але в UI не знайдено стабільного потоку запису назад у `/api/data/:tenantId` після add/edit/delete.

Наслідок:

- зміни можуть існувати лише в конкретному браузері
- різні адміністратори можуть бачити різний стан
- позиціонування як multi-tenant платформи частково не відповідає реальній моделі даних

Це `P0/P1`, залежно від того, чи система вже використовується кількома людьми.

### 4. Бекенд довіряє даним, які надсилає браузер

Ендпоінти:

- `POST /api/chat`
- `POST /api/reports/generate`

очікують `facultyList` прямо з клієнта і використовують його як джерело істини.

Наслідок:

- клієнт може підмінити дані перед відправкою
- бекенд не гарантує, що відповіді/звіти будуються на реальному tenant dataset

Для production це неправильний trust boundary.

### 5. Є XSS-ризик у відображенні AI/Markdown HTML

Небезпечні місця:

- `components/ChatInterface.tsx`
- `components/ReportGenerator.tsx`

Там використовується `dangerouslySetInnerHTML`, а HTML перед цим не санітизується.

Наслідок:

- потенційне виконання шкідливого HTML/JS у браузері
- особливо ризиковано, бо джерелом є AI-generated content і markdown parsing

Це `P0/P1`.

## Важливі, але не критичні проблеми

### 1. Фронтенд надто монолітний

Розміри ключових файлів:

- `App.tsx` — 1146 рядків
- `components/ReportGenerator.tsx` — 796 рядків
- `components/Dashboard.tsx` — 642 рядки
- `components/FacultyList.tsx` — 510 рядків

Наслідок:

- складно розвивати
- важко тестувати
- висока ймовірність регресій при змінах

### 2. Слабка строгість TypeScript на фронтенді

У фронтендному `tsconfig.json`:

- немає `strict`
- дозволено `allowJs`

Окремо по коду є багато `any` і ручних приведень типів.

Наслідок:

- помилки виявляються пізно
- складніше проводити безпечний рефакторинг

### 3. Відсутні тести й linting

У кореневому `package.json` немає:

- `test`
- `lint`

Наслідок:

- немає автоматичного захисту від регресій
- якість змін повністю залежить від ручної перевірки

### 4. Конфігураційний дрейф і залежність від CDN

У `index.html` підтягуються з CDN:

- Tailwind через `cdn.tailwindcss.com`
- `mammoth`
- `PapaParse`
- `xlsx`
- `pdf.js`
- `marked`
- `jspdf`

Також присутній import map для частини пакетів.

Наслідок:

- середовище працює не повністю через локально керовані npm-залежності
- поведінка залежить від зовнішніх CDN
- гірша відтворюваність build/runtime

### 5. У Vite-конфігурації інжектиться `GEMINI_API_KEY` у клієнт

У `vite.config.ts` через `define` пробрасывається:

- `process.env.API_KEY`
- `process.env.GEMINI_API_KEY`

При тому, що звіти й чат уже ходять на сервер.

Наслідок:

- ризик випадкового витоку ключа у фронтендний bundle
- зайва й небезпечна конфігурація

Це варто прибрати.

## Функціональні обмеження поточної реалізації

### 1. Генерація звітів може бути неповною

У `server/src/reportService.ts`:

- standard report бере лише перші 20 faculty через `slice(0, 20)`
- template mode обрізає context до 60000 символів

Наслідок:

- на більших tenant-ах звіти можуть бути неповними або неточними
- це особливо небезпечно, бо помилка неочевидна для користувача

### 2. Історія чату й звітів не ізольована по tenant-ах

У localStorage використовуються загальні ключі:

- `chat_history`
- `saved_reports`
- `api_keys`

Наслідок:

- при роботі з різними tenant-ами в одному браузері дані можуть змішуватись

### 3. Великий фронтендний bundle

Build фронтенду показав chunk близько:

- `818 kB` before gzip

Це не ламає систему, але вже впливає на:

- швидкість завантаження
- mobile UX
- масштабованість UI

## Загальна оцінка

### Рівень зрілості

Орієнтовно:

- продуктова ідея: висока зрілість
- backend architecture: середня/висока
- frontend maintainability: середня/низька
- security for multi-tenant production: низька
- operational readiness: середня

### Чесна оцінка

ResearchIQ зараз добре виглядає як:

- робочий demo
- внутрішній pilot
- MVP для обмеженого кола користувачів

Але ще не виглядає готовим як:

- повноцінна безпечна multi-tenant SaaS-система
- production-сервіс для приватних інституційних даних без доопрацювань

## Рекомендований порядок наступних кроків

### P0 — робити першими

1. Закрити серверний доступ до tenant-даних справжньою авторизацією.
2. Виправити логіку зміни admin password.
3. Прибрати довіру до `facultyList` з браузера для chat/report endpoints.
4. Додати санітизацію HTML перед рендером AI/Markdown контенту.
5. Забрати `GEMINI_API_KEY` із фронтендної конфігурації.

### P1 — робити одразу після цього

1. Зробити сервер єдиним джерелом істини для faculty CRUD.
2. Ізолювати localStorage-ключі по tenant-ах.
3. Розбити `App.tsx` і `ReportGenerator.tsx` на менші модулі.
4. Додати базовий linting і мінімальний smoke-test flow.

### P2 — покращення якості

1. Перенести CDN-залежності у керовані npm-пакети.
2. Увімкнути строгіший TypeScript на фронтенді.
3. Оптимізувати bundle через code-splitting.
4. Покращити великий report pipeline для великих tenant-ів.
5. Додати підтримку альтернативного LLM-провайдера `OpenRouter` з серверним auto-discovery безкоштовної моделі.

## Окремий feature-напрям: OpenRouter + auto-discovery free моделей

Цю інтеграцію варто планувати після закриття базових `P0/P1`, бо вона розширює гнучкість платформи, але не повинна випереджати безпеку й стабілізацію multi-tenant архітектури.

### Ідея

Додати в бекенд другий LLM-провайдер поряд із Gemini:

- `Gemini` як поточний основний провайдер
- `OpenRouter` як альтернативний або fallback-провайдер

При цьому вибір моделі для OpenRouter має відбуватись автоматично через серверний сервіс discovery, а не вручну жорстко зашиватись у код.

### Підтверджене джерело для auto-discovery

Станом на `2026-04-23` endpoint:

- `https://shir-man.com/api/free-llm/top-models`

повертає:

- `models[0].id` — поточну рекомендовану free-модель
- `fallback.id` — запасний варіант `openrouter/free`
- `baseUrl` — `https://openrouter.ai/api/v1`

Станом на `2026-04-23` у відповіді `models[0].id` був:

- `inclusionai/ling-2.6-flash:free`

А fallback-модель:

- `openrouter/free`

### Як це краще реалізувати

1. Додати на бекенді абстракцію `LLM provider layer` замість прямої прив'язки лише до Gemini.
2. Винести вибір провайдера в конфіг:
   - `LLM_PROVIDER=gemini|openrouter|auto`
3. Додати серверні env-параметри:
   - `OPENROUTER_API_KEY`
   - `OPENROUTER_MODEL`
   - `OPENROUTER_DISCOVERY_URL`
   - `OPENROUTER_FREE_MODEL_CACHE_TTL`
4. Реалізувати discovery-сервіс, який:
   - читає `models[0].id` із `https://shir-man.com/api/free-llm/top-models`
   - кешує результат на певний TTL
   - у випадку помилки або порожньої відповіді падає на `fallback.id`
   - якщо і це недоступно, використовує локально заданий backup model id
5. Тримати весь discovery і доступ до API-ключів тільки на бекенді.
6. Не дозволяти фронтенду напряму обирати model id для OpenRouter без серверної валідації.

### Де саме це буде корисно в поточному проєкті

- для `AI chat`
- для `report generation`
- для можливого майбутнього cheap/fallback execution mode
- для зниження залежності від одного вендора

### Практична логіка вибору

Рекомендований порядок:

1. Якщо `LLM_PROVIDER=gemini`, використовувати поточний Gemini flow.
2. Якщо `LLM_PROVIDER=openrouter`, використовувати OpenRouter.
3. Якщо `LLM_PROVIDER=auto`, то:
   - пробувати Gemini як primary, якщо ключ налаштований і сервіс здоровий
   - інакше переходити на OpenRouter
4. Для OpenRouter:
   - спочатку брати `OPENROUTER_MODEL`, якщо він явно заданий
   - інакше брати `models[0].id` із discovery endpoint
   - якщо discovery недоступний, падати на `openrouter/free`

### Ризики, які треба врахувати

- список free-моделей є динамічним і може змінюватись щодня
- безкоштовні моделі можуть мати нестабільні rate limits
- не всі free-моделі однаково добре працюють із tools / structured outputs
- для research/report задач може знадобитися окреме правило вибору моделі під сценарій

### Статус реалізації

Станом на `2026-04-23` цей напрям уже частково реалізований:

- на бекенді додано окремий `LLM provider layer`
- підтримано `LLM_PROVIDER=gemini|openrouter|auto`
- додано OpenRouter free-model discovery з кешем
- додано fallback на `openrouter/free`
- додано окремі override для chat і report use case

Тобто цей пункт більше не є суто `P2`-ідеєю, а вже перейшов у фазу operational hardening і deployment-настроювання.

### Бажаний результат інтеграції

Після реалізації система повинна вміти:

- працювати і з Gemini, і з OpenRouter
- автоматично знаходити актуальну free-модель для OpenRouter
- безпечно деградувати до `openrouter/free`
- не виносити ключі чи model selection logic у фронтенд
- дозволяти ручний override через env-конфігурацію

## Фінальний висновок

Проєкт уже має хорошу інженерну основу і реальну прикладну цінність. Найсильніша сторона — бекенд і AI-архітектура. Найслабші місця — безпека multi-tenant режиму, довіра до клієнта і надто монолітний фронтенд.

Якщо закрити `P0` і `P1`, цей проєкт можна досить реалістично довести до production-ready стану без повного переписування.
